04 — 기술

어떻게 만들었나

별도 개발 없이, Claude Code에서 프롬프팅만으로 구현했습니다. MRT 상품 URL 하나를 넣으면 HTML이 생성되는 /make-pdp 스킬을 만들어 구동했습니다.

프롬프팅 기반 파이프라인
URL 입력 → Chrome 크롤링 → __NEXT_DATA__ 추출 → 카테고리 판별 → 템플릿 매칭 → HTML 자동 생성
이 전체 과정을 AI 프롬프팅으로 설계하고 구동했습니다. 코드 직접 작성 0줄.
URL → Chrome DevTools (크롤링) → __NEXT_DATA__ 추출 → 카테고리 자동 판별
→ 템플릿 매칭 → 블록 데이터 매핑 → 이미지 매핑 (CDN URL) → HTML 생성 → meta.json 저장 → 브라우저 프리뷰
구동 모델
Claude Opus 4
카피 품질과 블록 정확도가 가장 높았습니다. Sonnet은 비용 1/5 대안.
출력 토큰
~30K / PDP
inline CSS 기준. external CSS 분리 시 ~17K (45% 절감).
비용
~$1.5 / PDP
Opus 기준. CSS 분리 시 ~$1.0. Sonnet이면 ~$0.3.
생성 시간
3 – 5분
크롤링 30초 + 정제 30초 + HTML 생성 2-4분.
데이터 소스
__NEXT_DATA__
SSR 데이터. logMeta + partitions에서 전체 정보 추출.
CSS 전략
inline → external
external 분리 시 토큰 절감 + 스타일 일관성 확보.
Opus vs Sonnet 비교 결과

LA 코리아타운 투어를 양쪽 모델로 생성 비교:

  • Opus: 카피라이팅 품질이 높고, 블록 구성이 정확함. 비용 ~$1.5.
  • Sonnet: 구조는 유사하나 디테일한 카피 품질이 다소 떨어짐. 비용 ~$0.3.
  • Opus 버전 vs Sonnet 버전
입력/출력 규격

입력: /make-pdp <URL>

  • --exclusive — PB Exclusive 카테고리 강제
  • --category ticket — 카테고리 수동 지정
  • --images ./images/ — 이미지 폴더 지정

출력:

  • output/{slug}.html — 자체 완결형 PDP HTML
  • output/{slug}.meta.json — 생성 메타데이터

08 — 레슨런

4일 · 13 Phase · 14 PDP에서 배운 것

3/10 스키마 설계부터 3/13 배치 생성까지, 실패와 전환을 반복하며 쌓인 핵심 교훈입니다.

가장 중요한 발견
같은 규칙인데 매번 다른 결과가 나온다면, 규칙이 아니라 선택지가 문제다. 텍스트 색상 7단계를 4단계로, 간격을 토큰으로 줄이자 생성 일관성이 확보되었다.
Phase 4–6 · 파일럿
같은 규칙, 다른 결과가 나오는 원인을 찾아라
동일한 토큰 체계인데 6개 파일럿의 텍스트 색상, 간격, 이미지 높이가 모두 달랐습니다. 원인은 선택지가 너무 많은 것 — 텍스트 색상 7단계를 4단계로, 간격을 6종 토큰으로 줄이자 일관성이 확보되었습니다.
토큰 v3: 값 + 용도 + 제약
Phase 9 · 첫 자동 생성
이미지는 자동 배치되지만, 맥락 매칭은 별개 문제
시드니 헬리콥터 상품에서 CDN 이미지 98장 자동 추출에 성공했지만, 12장을 블록에 배치할 때 순서만으로는 맥락이 맞지 않습니다. “20분 하늘 비행!” step에 지상 사진이 들어가는 등의 문제.
이미지 선택은 수동 검증 필수
Phase 10 · CVR 기반 재설계
데이터가 설계를 이끌면 설득력이 달라진다
ACTIVITY가 CVR에 가장 민감한 카테고리(9/9개월 양의 효과)인데 시각 비중이 최저(33%)였습니다. CVR 데이터 기반으로 2-Zone 아키텍처를 설계, 시각 비중 33% → 64%로 전환.
CVR 임계점이 블록 배치를 결정
Phase 11 · DS 비교
“안 바꾸면 어떤 문제가 생기는가?”를 먼저 묻는다
MRT Design System과 11개 항목을 비교했지만, 채택은 5개뿐이었습니다. 문제가 없으면 바꾸지 않는다 — 이 기준이 불필요한 변경을 6개 걸러냈습니다.
11개 검토, 5개 채택, 6개 제외
Phase 13 · 비용 최적화
CSS 분리가 가장 효율적인 비용 절감이었다
Opus vs Sonnet 비교 결과, Sonnet 카피는 “수정”이 아닌 “전면 교체” 수준이었습니다. 대신 CSS 외부 분리만으로 출력 토큰 45% 절감, PDP당 비용 $3 → $1.5.
생성 비용 50% 절감
Phase 12 · 배치 생성
Big Bang 대신 점진적 전환이 안전하다
v3 토큰으로 일괄 전환 대신, deprecated 마커를 붙여 기존 파일럿을 보존했습니다. 파일럿은 “골든 샘플”이라 함부로 건드릴 수 없고, 자연스럽게 전환하는 것이 리스크가 적습니다.
기존 보존 + 신규부터 적용
Phase 3 · 첫 PoC
스펙의 빈틈은 실제 상품에서만 보인다
나오시마·구마모토 PDP로 첫 하이브리드 렌더링을 검증했을 때, 설계 단계에서 미처 예상하지 못한 블록 구성 문제가 드러났습니다. 문서로 된 스펙은 첫 PoC가 발견하지 못한 것을 발견하지 못합니다.
PoC는 스펙 검증기
Phase 7–9 · 자동 생성
상품 데이터는 이미 JSON으로 정제되어 있었다
별도 API 없이 MRT 상품 페이지 내 __NEXT_DATA__에 이미지 98장, 가격, 코스, 리뷰까지 모두 담겨 있었습니다. 크롤링이 아니라 JSON 파싱으로 E2E 자동 생성이 가능했습니다.
__NEXT_DATA__ = 숨겨진 API
Phase 8 · 템플릿 추출
파일럿과 템플릿은 역할이 다르다
실제 데이터가 들어간 파일럿은 “무엇이 가능한지”를 보여주는 골든 샘플로 보존했고, 빈 플레이스홀더 구조로 분리한 템플릿은 자동 생성의 재료로 사용했습니다. 같은 파일 하나로 두 역할을 함께 하면 둘 다 망가집니다.
골든 샘플 ≠ 재사용 템플릿

09 — 통합

각자의 장점을 모아 방향 잡기

TF 멤버 각자가 독립적으로 새로운 접근을 시도하고 있습니다. 이 리뷰에서 각자의 결과물과 장단점을 공유하고, 최종 방향을 함께 정합니다.

리뷰 진행 방식
1. 각자의 접근 공유: 어떤 방식으로 시도했고, 어떤 결과가 나왔는지 발표
2. 장단점 비교: 블록 구성, 카피 품질, 디자인 완성도, 자동화 가능성 등 다각도 비교
3. 방향 합의: 각 접근의 장점을 살려 하나의 방향으로 수렴
4. 역할 분담: 합의된 방향에 따라 다음 단계 작업을 나눈다
결과물 통합 시 meta.json 규격
